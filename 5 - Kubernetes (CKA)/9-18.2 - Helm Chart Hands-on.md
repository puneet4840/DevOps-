# Helm Chart Hands-On

<br>

### Helm kya hota hai?

Helm basically Kubernetes ka package manager hai.

Jaise:
- Linux mein → ```apt```, ```yum```.
- Node.js mein → ```npm```.
- Python mein → ```pip```.

Waise hi Kubernetes mein → **Helm**.

<br>
<br>

### Problem kya thi (Helm kyun bana?)

Socho tum manually Kubernetes par koi application deploy kar rahe ho:

To tum khud se in files ko create karke inke ander configuration likhoge:
```
Deployment.yaml
Service.yaml
Ingress.yaml
ConfigMap.yaml
Secret.yaml
```

Agar 1 ya 2 applications ke liye ye files likhni ho to manually koi problem nahi hai.

Lekin agar ek bade application ki 10 microservices ho to:
- Har ek microservice ke liye 4–5 YAML likhi padengi.
- Har environment (dev, qa, prod) ke alag configs likhni padenge.

To kya hoga?
- Itni yamls ko manaually manage karna bohot problematic ho jayega.
- Copy-paste karte mein errors ho sakte hain.
- Version control mess.
- Rollback mushkil.

Isi problem ko solve karta hai **Helm**.

<br>
<br>

### Helm ka core concept

Helm ke through agar tumko kubernetes par application deploy karna hai to tum ek helm chart create karte ho.

Jab tum helm ki command se ek helm chart create karte ho to, helm apko ek folder create karke deta hai jiske ander kubernetes ki configuration files hoti hain.

Vo folder structure kuch esa hota hai:
```
mychart/
  Chart.yaml
  values.yaml
  templates/
    deployment.yaml
    service.yaml
```

Jo files aap application deploy karne ke liye create karte ho, vo helm apko khud create karke deta hai.

Ab in files ko samjho:

- Chart.yaml: Ye ek meta-data file hoti hai helm chart ki, iska matlab hai ki is file mein chart ki information store rehti hai, jaise
```
apiVersion: v2
name: nginx
description: A Helm chart for Kubernetes
version: 0.1.0
```

- values.yaml: Is file mein hum values store karte hain jo baad mein dynamically templates mein use hoti hain, jaise:
```
replicaCount: 2
image:
  repository: nginx
  tag: latest
service:
  type: ClusterIP
```

- templates: Ye dynamic yaml files hoti hain, jb tum application ko manually deploy karne ke liye create karte ho jaise, deployment.yaml, service.yaml. To helm in files ko khud create kar deta hai aur saath mein syntax bhi likh deta hai, tum in files ko haath lagane ki bhi jarurat nhi hoti hai, in files ke ander dynamic values lene ka syntax likha hota hai.

Example:

```deployment.yaml```:
```
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-nginx
spec:
  replicas: {{ .Values.replicaCount }}
```

```.Values.replicaCount``` values.yaml file se is yaml file mein aayega. Jo value tumne replica ki ```values.yaml``` file mein likhi hogi, vo yahan aa jayegi.

Ab tum dekh rahe ho ki ```{{ .Release.Name }}-nginx``` likha hua hai, ye release kya hoti hai.

<br>

**Release kya hota hai?**:

Jab tum chart ko install karte ho:
```
helm install myapp mychart
```
To ek instance create hota hai → usko bolte hain Release. Matlab uper command mein ```myapp``` likha hai to is naam se hi application create hoga, to isi ko release kehte hain.

To chart create hone ke baad hi ye ```myapp``` naam ```{{ .Release.Name }}-nginx``` release.name ki jagah par aa jayega.

### Helm ko use kaise kare (Matlab application kaise deploy kare)

Helm se application deploy karne ka ek flow hota hai, jaise:

- Helm chart create karo.
  - Isse tumko ek directory structure milega jisme ```charts.yaml```, ```values.yaml``` and templates hongi.
- Fir ```values.yaml``` file mein apne according values enter karo.
- Helm lint se syntax check karo.
- Agar production par ho to dry-run karke check dekho ki chart koi error to nhi de rha.
- Fir last mein chart ko install kar do.
- Chart install hone par apka application kubernetes par deploy ho jayega.
- Agar chart kisi ke saath share karna hai to chart ko package kar sakte ho isse chart ki ```.tar``` compress file create ho jaye, jisko share kar sakte hain.

Agar apka application pehle se helm ke thorough chal rha hai aur apko koi configuration change karna hai to helm upgrade karo.


<br>
<br>

Ab steps ko detail mein samjhte hain:

<br>

**STEP-1: Create a Helm Chart**:

Tum application ko directly deploy nhi karte ho,

Tu ek template system banata hai jise Helm me bolte hain: **Chart**.

Syntax:
```
helm create <chart-name>
```

Example:
```
helm create myapp
```

Ab kya hua internally?
- Helm ne ek ready-made structure bana diya.
- Ye koi deployment nahi hai abhi.
- Ye sirf ek directory structure hai.

Tum ek folder ```myapp``` ke ander ye directory structure milega:
```
nginx/
├── Chart.yaml       # Metadata about the chart
├── values.yaml      # Default configuration values
├── templates/       # Folder containing Kubernetes YAML templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
|   ├── serviceaccount.yaml
|   └── hpa.yaml
```

<br>

**STEP-2: Edit values.yaml file**:

Helm chart yani ek folder structure create hone ke baad tumko ab ```values.yaml``` file mein application ke according values deni hoti hain.

```values.yaml``` file ko bina modify kare, tumhara chart ek generic template hai.

Tum apni application ke according ```values.yaml``` ko modify karte ho.

Jaise:
- container image dena.
- port dena.
- name dena

Ye sab information tum apni application ke according dete ho.

Example:
```
replicaCount: 1
image:
  repository: nginx
  pullPolicy: IfNotPresent            
```

<br>

**STEP-3: Check chart Errors**:

Ab chart complete banne ke baad tum check karoge ke chart mein error ya koi issue to nahi hai.

Syntax:
```
helm lint <chart-name>
```

Example:
```
helm lint myapp
```

Output:

Agar chart mein koi error nahi hota to ye esa output deta hai.
```
==> Linting myapp
[INFO] Chart.yaml: icon is recommended

1 chart(s) linted, 0 chart(s) failed
```

<br>

**STEP-4: Dry-run the chart**:

Chart ko deploy karme se pehle api server se validate karwana.

Chart ko deploy karne se pehle ek baar dry run kar lo, jisse chart ko production pe deploy karne se pehle agar koi error hoga to vo error deploy hone se phle pta lag jayega.

Iska matlab ye command helm ko chart ko api server ko bhejti hai aur chart ko validate karti hai. dry-run mein chart ka koi bhi deployment nhi hota hai, sirf validation hota hai.

Syntax:
```
helm install <Release-Name> <Chart_name> --dry-run -debug
```

Example:
```
helm install myapp myapp --dry-run -debug
```

Yahan humare paas 2 options hote hain:
- Ya to ```helm install myapp myapp --dry-run``` commnad chalo.
- Ya for ```helm install myapp myapp --dry-run -debug``` command chalao.

Ye dono commands hi dry-run ke liye use hoti hain lekin inme difference hai. Chalo samjhte hain.

<br>

Pehle ```helm install myapp myapp --dry-run``` samjhte hain:
- Jab tum helm install --dry-run chalate ho, toh Helm sabse pehle templates aur ```values.yaml``` file ko merge karta hai aur templates mein dynamic code ki jagah value put karta hai jisse yamls files yani templates render ho jati hain, fir tumhare rendered YAML manifests ko Kubernetes API Server ke paas bhejta hai.

- API Server check karta hai:
  - Syntax Validation: Kya ye YAML Kubernetes ke schema ke mutabik sahi hai?
  - Resource Discovery: Kya tum jo Ingress version use kar rahe ho, wo cluster support karta hai?
  - Kubernetes API check karta hai ki kya tum jo resource banane ki koshish kar rahe ho, wo pehle se wahan exist toh nahi karta (conflict check).
 
API server is step mein chart par "Valid" ya "Invalid" ka thappa lagata hai.

Sab verify hone ke baad, API server use memory se uda deta hai, ETCD (database) mein save nahi karta.

<br>

Ab ```helm install myapp myapp --dry-run -debug``` samjhte hain:
- Agar tum ```--dry-run``` ke saath ```--debug``` laga dete ho, toh tum Helm ko "X-Ray mode" mein daal dete ho.
- ```--debug``` ke saath, tumhe Internal Helm Logic aur Server Response bhi dikhta hai.

- ```--debug``` lagane se ye 3 extra cheezein milti hain jo simple dry-run mein nahi hoti:
  - User-Supplied Values: Ye section dikhata hai ki tumne manually (```--set``` ya -f values.yaml se) kaun-kaun si values pass ki hain.
  - Computed Values: Ye sabse bada farq hai. Ye dikhata hai ki default values.yaml aur tumhare overrides ko merge karne ke baad Final Dictionary kya bani jo templates mein use ho rahi hai.
  - Client-Server Communication: Agar cluster se connect hone mein deri ho rahi hai ya authentication fail ho raha hai, toh --debug tumhe exact connection logs dikhayega.
 
Ek tarah se ```-debug``` tumko detailed logs dikhata hai.

<br>

**STEP-5: Chart Package karna**:

Packaging ka matlab hai apne poore chart directory (jisne dher saari files hain) ko ek single, compressed, aur versioned file mein convert karna. Iska maqsad hota hai chart ko share karna ya kisi Helm Repository (jaise Artifact Hub ya Azure Container Registry) mein push karna.

Syntax:
```
helm package <chart-name>
```

Example:
```
helm package my-app
```
Output:
```
my-app-0.1.0.tgz
```

Jab tum package command chalate ho:
- Helm tumhare myapp folder ko scan karta hai aur ek compressed file banata hai jiska format hota hai: ```<chart-name>-<version>.tgz```

Example: ```myapp-0.1.0.tgz```

Packaging sirf "zip" karna nahi hai, ismein Helm kuch zaroori kaam karta hai:
- Version Validation: Helm ```Chart.yaml``` mein jaakar ```version``` field check karta hai. Agar version missing hai, toh package fail ho jayega.
- Dependency Management: Agar tumhare chart mein sub-charts hain (```charts/``` folder mein), toh packaging ke waqt Helm unhe bhi verify karta hai.
- Integrity Check: Helm ek Provenance file (```.prov```) bhi generate kar sakta hai (agar tum ```--sign``` use karo) jo ye proof hota hai ki ye package tumne hi banaya hai aur raste mein kisi ne ise "tamper" nahi kiya.

Tip:
- Packaging se pehle hamesha ```Chart.yaml``` ka version update karna mat bhoolna. Agar tumne code change kiya par version 0.1.0 hi rehne diya, toh naya package purane wale ko overwrite kar dega aur tum "Version Drift" mein fans jaoge.

<br>

**STEP-6: Chart ko Install karna**:

Ye step helm chart ko kubernetes cluster par deploy karta hai, jisse apka application run ho jata hai.

Jo jo yaml apne templates mein banai hoti hain unke pods create karta hai.

Jab tum helm install chalate ho, toh tum apne local chart ya package ko Kubernetes cluster ke andar ek Release mein convert kar dete ho.

<br>

**6.1** - The Basic Installation (Local Folder se):

Agar tum apne myapp folder ke bahar khade ho, toh tum direct folder se install kar sakte ho:
```
helm install my-release-name ./myapp
```

Example:
```
helm install myapp ./myapp
```
- ```my-release-name```: Ye tumhari deployment ka ek unique naam hai. Ek hi chart se tum ```my-app-dev``` aur ```my-app-prod``` do alag-alag releases bana sakte ho.
- ```./myapp```: Ye tumhare chart ka path hai

<br>

**6.2** - The Package Installation (.tgz file se):

Agar tumne helm package karke .tgz file bana li hai, toh tum use bhi install kar sakte ho:
```
helm install my-release-name myapp-0.1.0.tgz
```

Example:
```
helm install myapp myapp-0.1.0.tgz
```

<br>

**6.3**. The Repository Installation (Remote se):

Ye remote installation hota hai, agar kisi remote location par apka chart rakha hua hai to aap usko ese install kar sakte ho.

Jab tum companies mein kaam karoge, toh tum aise install karoge (jaise Bitnami ya Azure repo se):

Example:
```
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install my-db bitnami/mysql
```
- Pehle chart ko ```helm repo add``` se download karlo.
- Fir install karlo.

<br>

Ab process ko samjho ```helm install``` karne par kya hota hai:
- Validation: Pehle check karta hai ki ```Chart.yaml``` aur templates sahi hain ya nahi.
- Rendering: ```values.yaml``` aur templates ko merge karke final Kubernetes YAML banata hai.
- Transmission: Ye saara YAML Kubernetes API Server ko bhej diya jata hai.
- Resource Creation: Kubernetes API server check karta hai aur cluster mein Pods, Services, etc. create karna shuru karta hai.
- Release Creation: Release create hota hai. Release tumhare running app ka naam hota hai.
- State Storage (Important!): Helm cluster ke andar ek Secret banata hai (usi namespace mein). Is secret mein wo is "Release" ki poori history aur config save kar leta hai. Isko hum aage dekhenge.

Tips:
- Agar chart create karte time tumne namespace mention nahi kiya, toh ye default namespace mein jayega. Hamesha ```-n <namespace-name>``` use karo aur ```--create-namespace``` flag lagao agar namespace exist nahi karta.

Example:
```
helm install myapp ./myapp -n production --create-namespace
```

- Chart Install hone ke baad ```helm list``` chalao ye dekhne ke liye ki release "deployed" state mein hai ya nahi.

Installation ke baad kya?
- Jab installation khatam hoti hai, toh tumhe screen par ```NOTES.txt``` ka content dikhta hai. Wo tumhe batata hai ki:
  - App ka URL kya hai.
  - Password kaise nikaalna hai (agar koi secret hai).
  - Service ko access kaise karna hai.

<br>

**STEP-9: Chart ko Upgrade Karna**:

Agar aapne koi chart pehle se install kiya hua hai aur aap usmein koi change karna chahte hain—jaise image version update karna ya configuration badalna, to chart mein change karne ke baad chart ko kubernetes par apply karna chte ho chart ko bina delete kiye, to ```helm upgrade``` ka use hota hai.

Matlab apne changes ko kubernetes mein apply karne ke liye ye command use hoti hai.

Syntax:
```
helm upgrade <release-name> <chart-name>
```

Example:
```
helm upgrade myapp myapp
```

Ye changes apply karta hai bina delete kiye.

Jab tum ```helm upgrade``` chalate ho, toh Helm teen cheezon ko compare karta hai:
- Old Chart + Old Values: Jo abhi chal raha hai.
- New Chart + New Values: Jo tum ab apply kar rahe ho.
- Live State: Kubernetes cluster mein actually abhi kya chal raha hai.

Is comparison ke baad, Helm ek "Patch" generate karta hai aur sirf wahi changes Kubernetes API ko bhejta hai jo zaroori hain. Isse "Zero Downtime" achieve karna asaan ho jata hai. Aur pods ko rolling update se create karta hai.

Log aksar ```helm upgrade``` chala kar samajhte hain ki kaam ho gaya. Lekin Helm default mein sirf Kubernetes API ko "bol kar" nikal jata hai.
- Hamesha ```--wait``` flag ka use karein. Yeh tab tak exit nahi karega jab tak saare Pods 'Ready' state mein na aa jayein. Ye ```--wait``` flag CI/CD pipeline mein bohot useful hota hai, Isse aapke CI/CD pipeline ko pata chalega ki deployment sach mein success hui hai ya nahi.

Example:
```
helm upgrade myapp myapp --wait
```

<br>

Helm Upgrade Flags:

Sirf ```helm upgrade name chart``` likhne se kaam nahi chalta. Professional environment mein hum flags use karte hain. Har flag ka ek maqsad hai:
- ```--install```:
  - Kya karta hai: Agar tumne galti se aisi application ka upgrade chalaya jo pehle se installed hi nahi hai, toh Helm error de dega. Lekin agar tum --install add kar doge, toh Helm dekhega: "Accha, pehle se nahi hai? Toh main ise fresh install kar deta hoon."
  - Tip: CI/CD pipelines mein hamesha upgrade --install use karo taaki pehli baar mein bhi pipeline fail na ho.
 
- ```--values```:
  - Kya karta hai: Tumhari ```values.yaml``` file mein saari settings hoti hain (CPU, Memory, Replica Count). Upgrade ke waqt tum naya values file de kar configuration badal sakte ho.
  - Tip: Kabhi bhi production values ko command line par --set se mat badlo agar changes bahut zyada hain. Hamesha file use karo taaki record rahe.
 
- ```--atomic```:
  - Kya karta hai: Yeh ```--wait``` ke saath kaam karta hai. Agar upgrade fail ho gaya (manlo naye version mein bug hai aur Pod start nahi ho raha), toh ```--atomic``` apne aap pura system pichle working version par Rollback kar dega.

Best Practice Command:
```
helm upgrade my-app ./my-chart \
  --install \
  --values prod-values.yaml \
  --set image.tag=v2.0.1 \
  --wait \
  --atomic
```

### Helm State Storage Concept

Jab aap ```helm install``` ya ```helm upgrade``` chalate ho, to helm apke chart ki deployment ko kubernetes mein ek secret ke form mein store karta hai, isi ko state storage kehte hain.

Helm mein "State Storage" ka matlab seedha sa hai: Helm ko kaise yaad rehta hai ki aapne apne cluster mein kya-kya install kiya hai, Helm kahin na kahin toh likhta hoga na ki "Version 1 mein ye settings thi" ya "Abhi Version 2 chal raha hai." Jis jagah ye information store hoti hai Usi jagah ko State Storage kehte hain.

State storage ka matlab hai ki helm chart se deploy ki gyi application ki information ko store karna.

<br>

Jab tum ```helm install``` ya ```upgrade``` chalate ho, toh Helm usi Namespace mein jahan tumne app deploy ki hai, ek Secret create karta hai.

Is Secret ka naam kuch aisa hota hai: ```sh.helm.release.v1.<release-name>.v<revision-number>```:
- ```v1```: Helm ka internal storage version.
- ```<release-name>```: Jo naam tumne helm install ke waqt diya tha (e.g., ```myapp```).
- ```v<revision-number>```: Har baar jab tum upgrade karte ho, revision number badhta hai (v1, v2, v3...).

<br>

**Is Secret ke andar kya hota hai?**:

Agar tum is secret ko ```kubectl get secret <name> -o yaml``` karke dekhoge, toh tumhe ek lambi base64 encoded string dikhegi. Jab Helm ise decode karta hai, toh usmein ye 3 cheezein hoti hain:
- Chart: Poora ```.tgz``` package (templates, Chart.yaml sab kuch).
- Values: Wo saari values jo tumne ```values.yaml``` ya ```--set``` se di thin.
- Manifest: Final rendered YAML jo cluster par bhej diya gaya tha.

Ye tumhare helm chart ko store kar deta hai, jisse helm ko ye pta rahe ki usne konse version run kiye hain, kya config thi.

<br>

**State Storage Kyun Zaroori hai?**:

**1 - Rollbacks**:
- Jab tum ```helm rollback myapp 1``` chalate ho, toh Helm cluster ke database (etcd) mein jaakar ```v1``` wala secret dhoondta hai, wahan se purana manifest nikaalta hai, aur use cluster par phir se apply kar deta hai.
- Helm rollback ka matlab pichle version par wapas jana.

**2 - Drift Detection**:
- Jab tum ```helm upgrade``` karte ho, toh Helm current state (jo secret mein hai) aur naye code ko compare karta hai. Use pata hota hai ki "Pichli baar 3 replicas the, ab 5 karne hain."
- Jab aap upgrade karte ho, toh Helm check karta hai: "Pichli state kya thi?" aur "Ab user kya maang raha hai?". Jo difference hota hai, sirf wahi apply karta hai.

**3 - Rollback**: 
- Agar aapka naya deployment crash ho gaya, toh Helm purane "Secret" (State) ko padhta hai aur cluster ko wapas purani condition mein le aata hai.

**4 - History Management**:
- Helm default mein pichle 10 revisions ki history rakhta hai. Iska matlab cluster mein 10 secrets honge. Tum ise ```--history-max``` flag se control kar sakte ho. Yaani pichle 10 charts ki information store hoti hain.

<br>

Tip:
- Database-less Architecture: Helm ka apna koi alag se database nahi hota (jaise SQL ya MongoDB). Wo Kubernetes ko hi apna database ki tarah use karta hai. Iska fayda ye hai ki agar Helm client delete bhi ho jaye, toh bhi cluster ko pata hai ki kya deployed hai.

- Security Risk: Kyunki state storage Secrets mein hoti hai, isliye agar kisi ke paas tumhare namespace ke secrets padhne ki permission hai, toh wo tumhare chart ki saari configuration (including passwords) dekh sakta hai.

<br>

**Example: Secret Kaise Dekhe**:

Apne terminal par ye command chalao (agar tumne ```myapp``` install kiya hai):
```
kubectl get secrets -n <your-namespace> | grep helm
```

Tumhe wahan v1, v2 ki list dikhegi. Agar tum wo secret delete kar doge, toh Helm ke liye wo release "gayab" ho jayegi, bhale hi tumhare Pods cluster mein chal rahe hon!

<br>

**State Storage ko store karne ke Doosre Options, secrets ka alawa**:

Halaki Secrets default hain, par aap inka bhi use kar sakte ho (mostly configuration change karke):
- ConfigMaps: Pehle use hota tha, par ye secure nahi hai kyunki data encrypted nahi hota.
- SQL (PostgreSQL): Agar aapko Kubernetes ke andar data nahi rakhna, toh aap Helm ko external Database se connect kar sakte ho (Large scale setups mein).

### Helm Rollback

**Rollback Kya Hai?**:

Maano aapne Version 1 (V1) se Version 2 (V2) par upgrade kiya, lekin V2 mein bug nikal aaya aur pods crash hone lage.

Helm Rollback kya karta hai?
- Woh aapke State Storage (jo maine pehle samjhaya) mein jaakar purane Version (V1) ka Secret uthata hai aur uske saare manifests (YAMLs) ko dobara cluster par apply kar deta hai.

**Ye Kaam Kaise Karta hai?**:

Jab aap rollback command chalate ho, toh piche ye steps hote hain:
- Helm dekhta hai ki aapne kis "Revision" number par wapas jaana hai.
- Helm us purane Revision ka data (Chart + Values) Secret se nikaalta hai.
- Rollback karne par Helm purana data delete nahi karta, balki ek Naya Revision banata hai.

Example: Agar V1 (Good) -> V2 (Bad) hua, toh rollback karne par V3 banega, jo exactly V1 jaisa hoga.

**Basic Commands**:

Pehle history check karo:
```
helm history <release-name>
```

Isse aapko REVISION number dikhega. Phir rollback maaro:
```
# Pichle version par jaane ke liye
helm rollback <release-name>

# Kisi specific version (e.g. Revision 2) par jaane ke liye
helm rollback <release-name> <REVISIO number>

helm rollback <release-name> 2
```

**Critical Warning**:

Rollback use karte waqt ye 3 baatein dimaag mein ghusa lo:
- Database Migrations: Helm sirf Kubernetes objects (Pods, Services, ConfigMaps) ko rollback karta hai. Agar aapne V2 mein Database ka schema badal diya hai, toh Helm use wapas nahi karega. Wo aapko manually karna padega.
- Persistent Volumes (PV): Agar V2 ne data delete kar diya hai disk se, toh rollback karne se data wapas nahi aayega. Rollback sirf configuration wapas lata hai, data nahi.
- The "V3" Logic: Yaad rakhna, rollback karne ke baad aapka helm list mein version number badh jayega. Log confuse ho jaate hain ki "Main pichle version pe gaya toh number kam kyun nahi hua?" — Kyunki Kubernetes mein har ek change ek naya event hota hai.

<br>
<br>

### Helm Delete

Helm delete ka matlab hai ki apne deployed application ko kubernetes mein se delete karna. Matlab apna application delete karna.

Modern Helm mein ```helm uninstall``` use hota hai.

**Basic Command: Uninstall**:

Pehle command dekh lo (Helm v3 mein ```uninstall``` prefer kiya jata hai, ```delete``` uska alias hai):
```
helm uninstall <release-name> -n <namespace>
```

Example:
```
helm uninstall myapp -n prod
```

**Piche Kya Hota Hai?**:

Jab tum ye command chalate ho, toh Helm ye steps leta hai:
- Helm apne State Storage mein jaakar check karta hai ki ye release jo user uninstall karna chta hai is release ne kaun-kaun se resources (Pods, Services, Deployments) banaye the.

- Helm Kubernetes API ko bolta hai ki "Bhai, ye saari list delete kar do." Matlab is release ke saare resource kubernetes se delete ho jate hain

- By default, Helm us release ki poori History/Secrets ko bhi uda deta hai. Matlab delete ke baad tum ```helm rollback``` nahi kar sakte kyunki storage hi khatam ho gayi.

**"Keep History" ka Khel**:

Kabhi-kabhi tum chahte ho ki resources toh delete ho jayein, par Helm ko yaad rahe ki "Yahan pehle kuch tha." Uske liye use hota hai:
```
helm uninstall <release-name> --keep-history
```
Kyun? Taaki tum baad mein helm history dekh sako ya wapas usi configuration par aa sako. Agar tumne ye flag nahi lagaya, toh Helm ke liye wo release kabhi exist hi nahi karti thi.

**Kya-Kya Delete NAHI Hota?**:
- Namespaces: Agar Helm ne resources deploy kiye hain, toh wo sirf resources udayega. Agar tumne namespace manually banaya tha, toh wo delete nahi hoga.
  
- Persistent Volume Claims (PVCs): Ye sabse bada point hai. Helm default mein PVCs ko delete nahi karta (Data safety ke liye). Agar tumne app delete kar di, toh disk wahi padi rahegi aur tumhara bill fat-ta rahega. Tumhe manually delete karna padega: ```kubectl delete pvc <pvc-name>```.

- CRDs (Custom Resource Definitions): Agar tumne koi Operator install kiya hai, toh uske CRDs aksar delete nahi hote taaki doosre apps crash na karein.

**Common Mistake**:

Agar tumhara uninstall command atak jaye (pending state), toh iska matlab hai koi "Finalizer" Kubernetes resource ko delete hone se rok raha hai.

Solution: ```kubectl get <resource> -o yaml``` karke check karo ki kahan stuck hai.
