# Helm Package Concept

<br>

### Helm Package kya hota hai?

Helm package = tera chart ka compressed version (.tgz file)

Helm chart ko compress karke ek ```.tgz``` file mein convert kar dete hain, yahi helm package hota hai.

Matlab:
```
mychart/  →  mychart-0.1.0.tgz
```

Ye bilkul waise hi hai jaise:
- zip file
- tar file

<br>
<br>

### Why Helm Package use karte hain?

Soch tu ek company mein kaam kar raha hai.
- Tera chart ready hai.
- Ab tu kya karega?

Har baar folder share karega? (bad practice)

Package bana ke share karega.

Chart ko package isliye karna hota hai kyu package ke baad jo compressed file aati hai usko easily share kar sakte hain.

Mainly chart ko share karne ke liye package kiya jata hai.

<br>
<br>

### How to Package chart

**STEP 1: Chart ready hona chahiye**:

Example:
```
helm create myapp-chart
```

Structure:
```
myapp-chart/
  Chart.yaml
  values.yaml
  templates/
```

```Chart.yaml``` important hai:
```
name: myapp-chart
version: 0.1.0
```

Ye version package name mein use hota hai.

<br>

**STEP 2: Package create karna**:

Example:
```
helm package myapp-chart
```
Output:
```
myapp-chart-0.1.0.tgz
```

Ye ek compressed archive hai jisme:
- templates.
- values.
- metadata sab included hai.

<br>

**Package ke andar kya hota hai?**:

```
myapp-chart-0.1.0.tgz
  ├── Chart.yaml
  ├── values.yaml
  └── templates/
```

Same chart → bas compressed form.

<br>

**STEP 3: Package install kaise karte hain?**:

Direct install:
```
helm install myapp myapp-chart-0.1.0.tgz
```

Same result:
- deployment
- service
- pods

<br>

**Upgrade Kaise Karte hain**:

Example:
```
helm upgrade myapp myapp-chart-0.2.0.tgz
```

New version deploy.

<br>
<br>

### Chart Versioning ka role (VERY IMPORTANT)

Helm version automatically change nahi karta ❌

Tujhe khud manually version update karna padta hai ✅

<br>

**Version kahan defined hota hai?**:

File: ```Chart.yaml```
```
name: nginx-chart
version: 0.1.0
appVersion: "1.0"
```

<br>

**Do types ke version hote hain**:

1 - ```version``` (Chart Version):
- Ye Helm chart ka version hai.
- Packaging ke time use hota hai.

Example:
```
version: 0.1.0
```

Package banega:
```
nginx-chart-0.1.0.tgz
```

<br>

2 - ```appVersion``` (Application Version):
- Ye actual app ka version hai (nginx, node app etc.)

```
appVersion: "1.21"
```
- Ye sirf informational hai.
- Packaging pe effect nahi karta.

<br>

**Scenario samajho**:

**Step 1: First time package**:
```
version: 0.1.0
```

Chart package kiya:
```
helm package nginx-chart
```
Output:
```
nginx-chart-0.1.0.tgz
```

**Step 2: Tu change karta hai chart**:

Example:
- replicaCount change
- image change
- configmap add

BUT version change nahi kiya
```
version: 0.1.0
```

Aur chart package kar diya:
```
helm package nginx-chart
```
Output:
```
nginx-chart-0.1.0.tgz
```

To chart bhi same version par fir se package ho jayega.

Overwrite ho sakta hai.

<br>

**IMPORTANT RULE**:
- Har change ke baad version manually increase karna MUST hai.

**Step 3: Proper way (Best Practice)**:

Tu change karta hai chart → version bump karta hai:
```
version: 0.2.0
```

Chart package karta hai:
```
helm package nginx-chart
```

Output:
```
nginx-chart-0.2.0.tgz
```

Ab:
- clean versioning
- upgrade possible
- rollback safe

<br>
<br>

### Versioning Strategy (Industry Standard)

Helm uses Semantic Versioning (SemVer)

Format:
```
MAJOR.MINOR.PATCH
```

| Version | Meaning                |
| ------- | ---------------------- |
| 1.0.0   | initial stable release |
| 1.1.0   | new feature added      |
| 1.1.1   | bug fix                |
| 2.0.0   | breaking change        |

<br>
<br>

### Upgrade kaise work karta hai version ke saath?

Example:
```
helm upgrade myapp nginx-chart-0.2.0.tgz
```

Helm internally:
- new version detect karta hai.
- release history update karta hai.

<br>
<br>

### Rollback kaise kaam karta hai?

Example:
```
helm history myapp
```

Output:
```
REVISION   CHART VERSION
1          0.1.0
2          0.2.0
```

Rollback:
```
helm rollback myapp 1
```

### Helm mein --version ka concept

Jab hum pre-built application ko repo se chart ko install karte hain to command mein ```--version``` dete hain.

Ab isko samjho.

<br>

**Core Idea (Sabse pehle ye samajh le)**:

```--version``` ka use chart version select karne ke liye hota hai.

Ye Chart.yaml ke version field se related hai.

NOT related to:
- appVersion.
- Docker image version.

<br>

**--version ka matlab kya hota hai?**:

Example:
```
helm install myapp bitnami/nginx --version 13.2.20
```

Matlab:
- “Helm bhai, mujhe nginx chart ka exact version 13.2.20 install karna hai”.

Iska matlab hai ki jab hum repo se chart ko install karte hain to ye ```--version``` argument ka use karte hain, chart ka particular version install karne ke liye. Agar ye ```--version``` mention na kare to default latest version install hoga.

<br>

**Ye kaam kab karta hai?**:

ONLY jab tu Helm repository use kar raha hai.

Jab tum pre-built application ko kubernetes mein helm ke though deploy karte ho to tab ```--version``` use hota hai bss.

Example:

Repo mein multiple versions hote hain:
```
bitnami/nginx
  ├── 13.2.19
  ├── 13.2.20
  ├── 13.3.0
```

Tab tu choose kar sakta hai:
```
helm install myapp bitnami/nginx --version 13.2.20
```

<br>

**Internally kya hota hai?**:
- Helm repo ka ```index.yaml``` check karta hai.
- Matching version dhundta hai.
- Us version ka .tgz download karta hai.
- Usko install karta hai.

<br>

**Agar tu --version na de to kya hota hai?**:

Example:
```
helm install myapp bitnami/nginx
```

Default behavior:
- Helm latest version install karega.

Agar tum install karte time command mein ```--version``` nahi doge to helm repo se latest version ka package install karega.

Example:

Repo mein:
```
nginx chart versions:
13.2.18
13.2.19
13.2.20  ← latest
```

Case 1: Without version:
```
helm install myapp bitnami/nginx
```

Install Hoga;
```
13.2.20
```

Case 2: With version:
```
helm install myapp bitnami/nginx --version 13.2.19
```

Install hoga:
```
13.2.19
```

<br>

**Kya --version local chart pe kaam karta hai?**:

Example:
```
helm install myapp ./mychart --version 0.1.0
```

Answer:
- NO effect

Local helm chart par ```--version``` kaam nahi karta hai.

Kyun?
- Kyunki tu already specific chart folder de raha hai.
- Helm ko version choose karne ki need hi nahi.


Rule yaad rakh:
```
--version → sirf pre-built application ke repo charts ke liye
```

<br>

**Local package case**:

Agar tu ```.tgz``` use kare:
```
helm install myapp mychart-0.2.0.tgz
```

Yahan:
- version already file name mein defined hai
-```--version``` ki need nahi.


<br>

**Real DevOps Scenario**:

Tu production mein:
```
helm install myapp bitnami/mysql --version 9.4.1
```

Kyun?
- Stability maintain karni hai.
- Latest version risky ho sakta hai.
- Tested version use karna hai.

<br>

**Upgrade ke time bhi use hota hai**:
```
helm upgrade myapp bitnami/mysql --version 9.5.0
```

Ye explicitly upgrade karega us version pe.

## Local Helm Chart mein Apni yaml files add karke use karna.

Ye mujhe pata hai ki helm khud se template provide karta hai, agar mujhe kuch aur resources create karne ho jaise configmap, image pull secrete jesi chize to wo kaise hota hai.

Usko ab samjhte hain.

<br>

### Core Idea

Helm kuch bhi magical nahi karta. Jo bhi tu Kubernetes YAML mein likh sakta hai → wo sab Helm template bana ke create kar sakta hai.

Matlab:
- ConfigMap ✅
- Secret ✅
- imagePullSecret ✅
- PVC ✅
- Ingress ✅
- ANY Kubernetes resource ✅

Lekin helm sabhi resources create nhi karta hai, agar hum koi resource alag se chart ke through hi create karna ho to hum chart mein vo yaml bana sakte hain aur values define karke resource create kar skate hain.

<br>

### Approach kya hota hai?

```
1. Template banao (templates/)
2. Values define karo (values.yaml)
3. Helm install/upgrade karo
```

<br>

### Example 1: ConfigMap create karna

Suppose tumne already ek chart create kar liya hai.

Usme tum khud se ek configmap create karna chte ho.

**Step 1: Template file**:

📁 ```templates/configmap.yaml```
```
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-config
data:
  app.env: |
    ENV={{ .Values.env }}
    LOG_LEVEL={{ .Values.logLevel }}
```

**Step 2: values.yaml**:
```
env: production
logLevel: debug
```

**Step 3: Install**
```
helm install myapp mychart
```

Result:
- ConfigMap create ho jayega.
- dynamic values ke saath.

<br>

### Example 2: Secret create karna

📁 ```templates/secret.yaml```
```
apiVersion: v1
kind: Secret
metadata:
  name: {{ .Release.Name }}-secret
type: Opaque
data:
  username: {{ .Values.username | b64enc }}
  password: {{ .Values.password | b64enc }}
```

**values.yaml**
```
username: admin
password: mypassword
```

Helm kya karega:
- base64 encode karega.
- Kubernetes secret create karega.

Helm function:
```
| b64enc
```
