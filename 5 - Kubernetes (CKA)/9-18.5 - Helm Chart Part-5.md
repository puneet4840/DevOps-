# Manage Multiple Environments(Dev, Prod, Stage) using Helm

Yaha hum dekhenge ki helm ke through multiple environment jaise Dev, Stage, Prod ko kaise manage kiya jata hai.

<br>

### Problem Samajh Pehle

Maan le tu ek app deploy kar raha hai using Helm.

Ab tere paas 3 environments hain:
- Dev (testing ke liye).
- Staging (pre-production).
- Production (live users).

Ab problem:
- Har environment ka config alag hota hai.
- Lekin code same hota hai.

**Example Problem**:

| Config    | Dev    | Staging    | Prod    |
| --------- | ------ | ---------- | ------- |
| Replicas  | 1      | 2          | 5       |
| Image Tag | latest | stable     | v1.0.0  |
| Resources | low    | medium     | high    |
| DB URL    | dev-db | staging-db | prod-db |

To kya har baar alag chart banaoge?
- Nahi! Helm ka magic yahi hai — same chart + different values.

<br>
<br>

### Core Concept: values.yaml

Helm mein jo bhi configurable cheez hoti hai wo ```values.yaml``` mein hoti hai.

Example:
```
replicaCount: 1

image:
  repository: nginx
  tag: latest

resources:
  limits:
    cpu: "100m"
    memory: "128Mi"
```

Ye default values hoti hain.

<br>

**Solution: Multiple Values Files**:

Ab tu har environment ke liye alag values file bana sakta hai.

Structure:
```
mychart/
  values.yaml           # default
  values-dev.yaml
  values-staging.yaml
  values-prod.yaml
```

<br>

**Dev Values**:
```
replicaCount: 1
image:
  tag: latest
```

<br>

**Staging Values**:
```
replicaCount: 2
image:
  tag: stable
```

<br>

**Prod Values**:
```
replicaCount: 5
image:
  tag: v1.0.0
```

Iska main concept ye hai ki, alag environments ke liye humara chart to same hoga, lekin jo chart ko values provide karenge vo environment ke hisab se alag alag kar sakte hain.

<br>
<br>

### Deployment Kaise Karte Hain

Helm install ya upgrade karte time tu values file pass karta hai:

**Dev**:
```
helm install myapp ./mychart -f values-dev.yaml
```

**Staging**:
```
helm install myapp ./mychart -f values-staging.yaml
```

**Prod**:
```
helm install myapp ./mychart -f values-prod.yaml
```

Same chart → different configs → different environments ✅.

<br>
<br>

### Internal Working Samajh

Jab tu command chalata hai:
```
helm install myapp ./mychart -f values-prod.yaml
```

Helm kya karta hai internally:
- Chart load karta hai.
- Default ```values.yaml``` read karta hai.
- ```values-prod.yaml``` file se override karta hai.
- Templates render karta hai.
- Kubernetes YAML generate karta hai.
- Deploy karta hai.

<br>

**Template Example**:

```
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: nginx
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

Ye dynamic ho gaya environment ke according.

Ye ```deployment.yaml``` file ek generalized file hoti hai, jisme koi bhi ```values.yaml``` file ko use karke deployment create kar sakte hain. Isliye multiple values.yaml file use karne par bhi deployment hota hai.

<br>
<br>

### Next Level: Namespace-based Environments

Real world mein hum environments ko namespace se isolate karte hain:
- dev → ```dev-namespace```.
- staging → ```staging-namespace```.
- prod → ```prod-namespace```.

Example:
```
helm install myapp ./mychart -f values-dev.yaml -n dev
helm install myapp ./mychart -f values-staging.yaml -n staging
helm install myapp ./mychart -f values-prod.yaml -n prod
```

<br>

**Advanced Concept: Override Inline Values**:

Kabhi kabhi tu CLI se hi override karta hai:
```
helm install myapp ./mychart -f values-prod.yaml --set replicaCount=10
```

Highest priority hoti hai ```--set``` ki.

Iska matlab hai ki agar tumne install commnad mein ```values.yaml``` file di hai aur ```--set``` argument se value set kar rhe ho to, ese mein ```--set``` argumnet se value define hogi jabki tumne ```values.yaml``` file de rakhi ho, ```values.yaml``` file se value set nhi hogi.

```--set``` se hogi kyu ```--set``` argument ki highest priority hoti hai.

<br>
<br>

### Values Priority Order (IMPORTANT)

Helm values override priority:

- ```--set``` (highest).
- ```-f values-prod.yaml```.
- ```values.yaml``` (default).
