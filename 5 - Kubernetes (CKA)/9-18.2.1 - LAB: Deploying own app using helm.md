# LAB: Deploying Own Application on Kubernetes Using Helm Chart

Is slide mein hum ek khud ka application kubernetes par deploy karenge helm charts ke through.

<br>

### LAB GOAL

Tum kya banaoge:
- Ek Node.js app (Dockerized).
- Helm chart se deploy karoge.
- Proper structure + values.yaml + templating.
- Different environments (dev/prod).
- Ingress + Config + Secrets.
- Upgrade + rollback.
- Production best practices.

### STEP 1: Architecture samajh lo

Production mein flow kuch aisa hota hai:
```
Code → Docker Image → Push to Registry → Helm Chart → Kubernetes Cluster
```

Helm ka role:
- Kubernetes YAML ko dynamic banana.
- Environment-specific config manage karna.
- Versioning + upgrades handle karna.

### STEP 6: Deployment template samjho + modify karo

**templates/deployment.yaml**:
```
containers:
  - name: {{ .Chart.Name }}
    image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
    
    ports:
      - containerPort: {{ .Values.containerPort }}

    env:
      - name: APP_ENV
        value: {{ .Values.env.APP_ENV | quote }}

    resources:
      {{- toYaml .Values.resources | nindent 12 }}
```

Explanation of yaml:

- ```.Chart.Name```: Ye ```Chart.yaml``` file se tumhare project ka naam uthayega. Iska fayda ye hai ki agar tum chart rename karte ho, toh container ka naam apne aap update ho jayega.
- ```.Values.image...```: Ye sabse important part hai. Hardcoding ki jagah ye values ```values.yaml``` file se aati hain.

```
ports:
      - containerPort: {{ .Values.containerPort }}
```
- Yahan tum port ko centralize kar rahe ho. Agar kal ko tumhari application port 8080 ki jagah 3000 par shift hoti hai, toh tumhe poore template mein dhundhne ki zaroorat nahi, sirf ```values.yaml``` mein ek change kaafi hai.

```
env:
      - name: APP_ENV
        value: {{ .Values.env.APP_ENV | quote }}
```
- ```| quote```: Ye ek Helm function hai. Ye value ko double quotes mein wrap kar deta hai.
- Why it matters: Agar ```APP_ENV``` ki value true ya 123 hai, toh bina quotes ke YAML ise boolean ya integer samajh lega, jabki Kubernetes env value mein string expect karta hai. Ye crash hone se bachata hai.

```
resources:
      {{- toYaml .Values.resources | nindent 12 }}
```
- ```toYaml```: Ye poore ke poore block (CPU/Memory limits) ko values.yaml se utha kar yahan paste kar deta hai.
- ```nindent 12```: ```nindent``` ka matlab hai (New line Indent). Ye function 12 spaces add karta hai aur ek nayi line start karta hai.
  - Ye do kaam ek saath karta hai:
    - New Line: Ek nayi line start karta hai.
    - Spaces: Utne spaces add karta hai jitne tumne specify kiye hain.

Maano tumhari ```values.yaml``` aisi dikhti hai:
```
resources:
  limits:
    cpu: 100m
    memory: 128Mi
```

Aur tumhara template ye hai:
```
resources:
      {{- toYaml .Values.resources | nindent 12 }}
```
Peeche kya ho raha hai?
- ```toYaml``` ne ```values.yaml``` se wo data uthaya.
- ```nindent 12``` ne har line ke aage 12 spaces thok diye taaki wo resources: ke bilkul niche sahi jagah fit ho jaye.

Output Without Nindent:
```
resources:
      limits:
    cpu: 100m
```

Output with Nindent:
```
resources:
            limits:       # Yahan se yahan tak 12 spaces hain
              cpu: 100m
```


Dash (-) Ka Chakkar:
- Tumne dekha hoga {{- nindent ... }} mein ek dash hai. Iska matlab hai "Upar ki khali jagah (whitespace) ko khatam karo."
- Agar tum ```-``` nahi lagaoge, toh Helm template engine extra empty lines chor dega, jisse tumhara YAML "dirty" dikhega. ```{{-``` use karne se output ekdam clean aur compact aata hai.

Tumhe pata hona chahiye. Kubernetes YAML mein hierarchy hoti hai:
```
spec: (0 spaces)
  template: (2 spaces)
    spec: (4 spaces)
      containers: (6 spaces)
        - name: ... (8 spaces)
          resources: (10 spaces)
            limits: (12 spaces) — Isliye yahan 12 ki zaroorat padi.
```

Agar tumne ```nindent 10``` likh diya, toh limits seedha resources ke barabar aa jayega aur deployment CrashLoopBackOff mein bhi nahi jayegi, seedha Invalid YAML error degi.

### STEP 8: Ingress enable karo

values.yaml mein add karo
```
ingress:
  enabled: true
  className: nginx

  hosts:
    - paths:
        - path: /
          pathType: Prefix
```

templates/ingress.yaml check karo
- Helm already generate karta hai — bas enable karna hota hai.

Iska concept ye hai:

Jab tum ```helm create my-app``` chalate ho, toh Helm ek "Standard" ```ingress.yaml``` file tumhare ```templates/``` folder mein pehle se bana kar rakh deta hai. Wo file ek khaali dhancha (skeleton) hoti hai jo sirf tab active hoti hai jab tum ```values.yaml``` mein ```enabled: true``` karte ho.

```ingress.yaml``` kaisi dikhti hai?

Ye file complex lag sakti hai kyunki isme bohot saari loops aur conditions hoti hain taaki ye har tarah ke host aur path ko handle kar sake. Ek simplified version ye raha:
```
{{- if .Values.ingress.enabled -}}  # AGAR enabled true hai toh hi niche ka code chalega
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ include "my-app.fullname" . }}
  annotations:
    {{- toYaml .Values.ingress.annotations | nindent 4 }}
spec:
  ingressClassName: {{ .Values.ingress.className }}
  rules:
    {{- range .Values.ingress.hosts }} # Ye loop hai: values.yaml ke har host ke liye ek rule banayega
    - host: {{ .host | quote }}
      http:
        paths:
          {{- range .paths }} # Har host ke andar ke saare paths (like / or /api)
          - path: {{ .path }}
            pathType: {{ .pathType }}
            backend:
              service:
                name: {{ include "my-app.fullname" . }}
                port:
                  number: 80
          {{- end }}
    {{- end }}
{{- end }}
```

Jab tum ```values.yaml``` mein ye niche likha hua block add kar doge to helm cluster ke ander sirf ingress resource ko hi create karega. Jaise isme ```enabled: true``` likha hua hai, ye helm ko batata hai ki ingress resource create kardo.
```
ingress:
  enabled: true
  className: nginx

  hosts:
    - paths:
        - path: /
          pathType: Prefix
```

Lekin sirf ingress resource create karne se ingres setup nhi hoga, uske liye tumko ingress controller bhi install aur setup karna padega. Aaage padhe isi baare mein likha hai.

**Ingress Ka Setup (Step-by-Step)**:

Sirf YAML file hone se traffic tumhare app tak nahi pahunchega. Iske liye 3 cheezein honi chahiye:

Step A: Ingress Controller Install Karo:
- Agar cluster mein Controller nahi hai, toh tumhari ```ingress.yaml``` sirf ek bekar paper ka tukda hai.
- Nginx Ingress Controller install karo (Helm se hi kar sakte ho):
```
# Add Repo
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# Install Ingress Controller
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace

# Verify Karo
kubectl get pods -n ingress-nginx
```

Step B: values.yaml Update Karo:
- Ab tumko ```values.yaml``` file mein ingress ke liye values likhni hongi, jisme ```enabled: true``` likhe hoga.
- Jo tumne ```values.yaml``` mein ingress ki configuration likhi hai, wo bilkul sahi hai. Ye Helm ko batati hai ki:
  - Ingress resource banana hai (enabled: true).
  - nginx controller ko use karna hai.
  - Kyuki agar tum AKS par host karoge application aur tum public ip ko kisi host se bind nahi karoge to ese mein ye host dene ki koi jarurat nahi hai. Agar host hi hata doge to ye all host accept karega. Request kisi bhi host se aaye bas ingress rule match hona chiaye.
  - Lekin agar tum ek local minikube ya kind cluster use kar rhe ho to tumko ```values.yaml``` mein ye ```host: helm-app.local``` likhna padega. Kyuki browser ko batana hoga ki localhost use kar rhe ho.
 
Step C: Local Testing ke liye ```/etc/hosts``` (Agar tum local kind cluster use kar rhe ho to tab use karo):
- Kyuki ```helm-app.local``` koi real domain nahi hai, tumhare browser ko nahi pata ki kahan jana hai.
- Ppne computer ki ```/etc/hosts``` file mein ye line add karo:
  - ```127.0.0.1  helm-app.local``` (Replace 127.0.0.1 with your Ingress IP). Ye local host ke liye hai.

### STEP 9: Secrets + ConfigMap

**values.yaml**:
```
config:
  APP_NAME: "HelmApp"

secret:
  DB_PASSWORD: "mypassword"
```

**templates/configmap.yaml**:
```
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-config

data:
  APP_NAME: {{ .Values.config.APP_NAME }}
```

**templates/secret.yaml**:
```
apiVersion: v1
kind: Secret
metadata:
  name: {{ .Release.Name }}-secret

type: Opaque
data:
  DB_PASSWORD: {{ .Values.secret.DB_PASSWORD | b64enc }}
```

- ```| b64enc```: Ye Helm ka function hai jo tumhari plain-text value (e.g., Admin@123) ko Base64 string (QWRtaW5AMTIz) mein badal deta hai.

### STEP 11: Upgrade flow

Ab tum app mein kuch change karo jaise:
```
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('Hello from Helm Mera App 🚀');
});

app.listen(3000, () => {
  console.log('App running on port 3000');
});
```

Production ki jaga Mera likh do.

Fir docker image banake docker image ka naya version registry mein push karo aur uske tag ko ```values.yaml``` mein daalke upgrade kardo.

Image change karo:
```
tag: "v2"
```
Then:
```
helm upgrade my-app helm-app
```

Tumhara naya app deploy ho jayega.
