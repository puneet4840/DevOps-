# Helm Chart Pre-built application installation concept

Isse pehle wali slide mein humne dekha ki hum khud se application ko helm se chart se deploy karne ki baat kar rahe the. Lekin is slide mein hum dekhenge ki kaise helm chart se kuch pre-built application kubernetes mein kaise deploy karte hain.

<br>

### Helm se pre-defined ya pre-built apps ka concept kya hai?

Helm sirf tera custom chart nahi, balki ready-made charts (pre-built apps) bhi provide karta hai.

Inhe bolte hain: **Helm Charts from Repositories**

Jaise:
- Database.
- Monitoring tools
- Logging stack.
- CI/CD tools.

To isse pehle hum khud ke application ko helm chart ke through deploy kar rahe the, lekin hum helm chart ke through pre-built application jaise monitoring tools, logging stack, CI/CD tools aur database bhi install kar sakte hain.

Helm ka sabse bada fayda hi yahi hai ki ye Kubernetes ke liye ek App Store ki tarah kaam karta hai. Isme "Pre-built application" ka matlab hota hai Helm Charts, jo pehle se kisi expert ne likh kar public rakhe hue hain.

Jaise tum Android phone mein Play Store se WhatsApp ya Instagram install karte ho, waise hi tum Helm se Kubernetes mein pre-built apps (jaise MySQL, Nginx, Jenkins) install karte ho.

<br>

**Pre-built Application ka Basic Meaning**:

Pre-built application = already bana hua, ready-to-use software package.

Helm mein pre-built application ka matlab hota hai:
- Helm Chart jo already kisi application ke liye bana hua hai.

Example:
- Nginx server.
- MySQL database.
- Prometheus monitoring.
- Grafana dashboard.

Ye charts kisi developer ne already best practice ke saath create kiye hote hain.

Ye sab charts tumhe milte hain internet par repositories mein milte hain, tum bas download karte ho aur install karke application deploy karte ho.

<br>

**Kyun banaye gaye Pre-built Charts? (The Problem)**:

Socho tumhe ek PostgreSQL Database setup karna hai Kubernetes mein. Tumhe kya-kya likhna padega?

- Deployment/StatefulSet: Pods chalane ke liye.
- Service: Taaki doosre apps database se baat kar sakein.
- ConfigMap: Database ki configuration ke liye.
- Secrets: Password aur sensitive data ke liye.
- Persistent Volume Claim (PVC): Data ko permanent store karne ke liye.
- RBAC (Roles/ServiceAccounts): Permissions ke liye.

Ye kam se kam 500-1000 line ka YAML code ho jayega. Agar tumne ek choti si galti ki (jaise mount path galat de diya), toh tumhara database crash ho jayega ya data loss ho jayega.

Solution: Expert developers ne ye pehle se pre-built application ke files pehle se hi bana diye, jise hum Pre-built Chart kehte hain.

Tumko ye charts use karne ke liye phle tumko repositories se inko download karne padte hain, fir inko use karna padta hai.

<br>
<br>

### Repositories for downloading Helm Charts

Jab hum helm se pre-built application deploy karte hain to humko helm charts repositories se download karne padte hain, fir inko install karke use karna hota hai.

Pre-built application ka matlab hai ki vo application ho humko scratch se nhi create karne hote, jaise agar mujhe MySQL database kubernetes mein run karna hai to mujhe scratch se script likhkar usko use create karna padega, lekin pre-built charts se bss mujhe apne accodring values provide karke kubernetes par deploy karna hoga.

Pre-build charts ka yahi fayda hai ki mujhe scratch se kuch create nhi karna padega.

To expert developers ne pehle se hi pre-built files ko repositories pe upload kiya hua, jaha se tum download karte ho.

<br>

**Public Helm Repositories**:

Ye repos publically internet par available hoti hain, koi bhi user yahan se helm charts download karke use kar sakta hai.

- **Artifact Hub**:
  - Ye ek portal hai, Ye Google search ki tarah hai Helm charts ke liye. Ye koi storage nahi hai, balki ek Aggregator hai. Duniya bhar ki companies (Bitnami, HashiCorp, Grafana) apne charts yahan list karti hain.
  - Website: artifacthub.io
  - Kaise use karein: Yahan jao, app search karo (e.g., "Redis"), aur wo tumhe command bata dega ki kaunsa repo add karna hai.
 
- **Bitnami Charts**:
  - Ye ek repository hai jahan helm charts rakhe hote hain.
  - Agar tum ek DevOps engineer ho, toh tumhara sabse zyada waqt Bitnami ke charts ke saath guzrega. Inke charts bahut secure aur well-maintained hote hain.
 
- **Official Cloud Provider Repos**:
  - Har bada cloud provider apne tools ke liye charts maintain karta hai:
    - AWS: EKS related tools (Load balancer controller, Karpenter).
    - Azure: Azure Service Operator charts.
    - Google (GCP): Config Connector charts.
   
- **Infrastructure Tools Repos**:
  - Agar tumhe monitoring ya logging setup karni hai, toh unke official repositories use karo:
    - Prometheus/Grafana: prometheus-community repo.
    - HashiCorp: Vault aur Consul ke liye.
    - Jetstack: cert-manager ke liye (SSL certificates).
   
<br>

**Private Helm Repositories**:

Jab tum kisi company mein kaam karoge, toh tum public charts use nahi karoge apne code ke liye. Tum apna Private Registry banaoge. Uske liye ye tools use hote hain:
- **JFrog Artifactory**: Sabse popular enterprise tool.
- **ChartMuseum**: Ek open-source tool jo sirf Helm charts store karne ke liye banta hai.
- **OCI Registries** (Modern Way): Aaj kal log alag se server nahi banate. Wo Docker Hub, Azure Container Registry (ACR), ya GitHub Container Registry (GHCR) mein hi Helm charts ko "Container Image" ki tarah push kar dete hain.

Problem with Public charts:
- Sabko visible.
- Sensitive configs expose ho sakte hain.
- Company-specific apps share nahi kar sakte.

Isliye companies ya project khud ke charts banakar host karte hain private repositories mein.

<br>

**Best Practices**:

Jab bhi kisi portal se chart uthao, ye 3 cheezein check kiye bina deploy mat karna:
- Verified Publisher: Kya chart official company ne banaya hai (e.g., Redis by Bitnami)?
- Kube Version Compatibility: Check karo ki wo chart tumhare Kubernetes version ko support karta hai ya nahi.
- Last Updated: Agar koi chart 2 saal se update nahi hua, toh usme security vulnerabilities ho sakti hain. Use touch bhi mat karna.

### Helm Chart se Pre-Built Application Kaise Deploy Karte Hain

Yahan hum dekhenge ki pre-built application ko kaise deploy kiya jata hai.

**Step-1: Chart ko Search Karna**:

Sabse pehle tumhe ye pata lagana hai ki jo app tumhe chahiye uska official ya verified chart kahan hai.
- Portal: ```Artifact Hub``` par jao.
- Search bar mein likho "Nginx" ya "MongoDB".
- Hamesha dekho ki chart "Verified Publisher" (jaise Bitnami) ka ho.

<br>

**Step-2: Repository ko Add aur Update Karna**:

Kubernetes ko nahi pata ki Bitnami ya Google ke charts kahan rakhe hain. Tumhe wo "Address" (URL) apne local Helm client mein add karna padta hai.

Example:
```
helm repo add bitnami https://charts.bitnami.com/bitnami
```

Ye command kya karti hai?
- Remote repo ko local Helm config mein add karti hai.
- Ab tum us repo ke charts use kar sakte ho.

<br>

**Step-3: Repo update karna**:

Repo to download karli, kya lekin usme latest charts hone, isliye tumko repo ko update karna hoga, Isse tumhare paas latest charts ki list aa jati hai.

Example:
```
helm repo update
```

Kya hota hai:
- Latest charts ka index download hota hai.
- Version updates fetch hoti hain.

<br>

**Step-4: Chart search karna**:

Jo app tum deploy karna chte ho kya vo repo mein available hai, repo ko add karne ke baad fir se check karlo.

Example:
```
helm search repo nginx
```
Output:
```
bitnami/nginx
```

Kya hota hai:
- Repo ke andar available charts search karta hai.
- Tumhe chart ka naam milta hai.

<br>

**Step-5: Chart inspect karna**:

Chart ko install karne se pehle samajhna bahut zaruri hai, uski values ko dekhna important hai ki by default chart mein kon kon si values hain.

Example:
```
helm show values bitnami/nginx
```

Ye kya karta hai?
- Default configuration dikhata hai.

Example Values:
```
replicaCount: 1

service:
  type: ClusterIP
  port: 80
```

<br>

**Real Insight**:

Yahin pe DevOps engineer ka kaam start hota hai.

Tum decide karte ho:
- replicas kitne hone chahiye.
- service type kya ho (NodePort / LoadBalancer).
- resource limits kya ho.

Isliye tum khud ki ek custom ```values.yaml``` file banate ho.

<br>

**Step-6: Values ko Customize Karna**:

Pre-built app ko "As-is" install karna bewakoofi hai. Tumhe apne cluster aur application ke hisaab se settings badalni padengi.

Tum chart ki defalut values ko ek file mein save kar lete ho:

Example:
```
helm show values bitnami/nginx > my-values.yaml
```
Is command se saari values ```my-values.yaml``` file mein aa jayengi.

Ab is file ko open karo aur change karo:
- ```replicaCount: 2``` (agar 2 pods chahiye).
- ```service.type: LoadBalancer``` (agar bahar se access karna hai).
- ```resources.limits.cpu: "500m"``` (agar CPU limit karni hai).

Ya fir tum khud se ek custom ```values.yaml``` file create karlo, aur usme manually ek ek karke values define kardo.

Example:

Tum apna custom file banate ho:
```
vi my-values.yaml
```

Example:
```
replicaCount: 3

service:
  type: NodePort
  port: 80
```

<br>

**Step-7: Dry Run Chart**:

Chaer ko direct install karne se pehle hamesha check karo ki tumhara YAML sahi bana hai ya nahi. Isse cluster crash nahi hota.

Example:
```
helm install my-release bitnami/nginx --values my-values.yaml --dry-run --debug
```

Ye command chart ko install nahi karegi, aur na hi app ko deploy karegi. Simple ek demo deployment karti hai aur dikhati hai ki tumhari configuration se kya kya deploy hone wala hai.

<br>

**Step-8: Install Chart**:

Agar dry-run mein sab thik rehta hai to chart ko install karlo.

Example:
```
helm install my-app-nginx bitnami/nginx -f my-values.yaml
```

Ye command kya karti hai?
- Helm chart ko read karta hai.
- ```values.yaml``` apply karta hai.
- Kubernetes YAML generate karta hai.
- Cluster mein resources create karta hai.

<br>

**Step-9: Verification (Post-Deployment)**:

Install hone ke baad check karna zaroori hai ki "Pre-built" app sahi se chali ya nahi.

Status Check:

Example:
```
helm list
```
Ye command sabhi deploy releases ko dikhati hai, lekin ye command default namespace ki release ko dikhayegi.

Example:
```
helm list -A
```


Resource Check:

Example:
```
kubectl get all
```

Example:
```
kubectl get pods

kubectl get svc
```

<br>

### Tips for chart installation:

**Tip-1**:

Namespace: Kabhi bhi default namespace mein install mat karo. Hamesha ```-n my-namespace``` ```--create-namespace``` use karo.

Syntax:
```
helm install <release-name> <chart-name> --namespace <namespace-name> --create-namespace
```

Example:
```
helm install my-app bitnami/nginx -n prod --create-namespace
```

<br>

**Tip-2**:

Version Lock: Install karte waqt ```--version``` flag use karo. Agar tumne version fix nahi kiya, toh 6 mahine baad jab tum upgrade karoge, toh Helm automatically latest (aur shayad incompatible) version utha lega aur sab phat jayega.

Helm mein do versions hote hain: ```App Version``` (Software version) aur ```Chart Version``` (Infrastructure version). Jab tum ```--version``` nahi dete, toh Helm latest tag uthata hai.

Maano tumne ```Chart V1.0``` pe install kiya jo Kubernetes 1.25 support karta tha. Upgrade ke time Helm ne ```Chart V2.0``` utha liya jo sirf Kubernetes 1.28+ support karta hai. Tumhara deployment turant crash ho jayega.

DevOps pipelines mein version lock karna mandatory hai taaki Environment drift na ho (Dev aur Prod mein same version chale).

Example:
```
helm install my-nginx bitnami/nginx --version 15.1.0
```

<br>

**Tip-3**:

Uninstall Clean-up: Yaad rakhna, ```helm uninstall``` karne se tumhare PVCs (Storage) delete nahi honge. Unhe manually delete karna padega warna cloud bill badhta rahega.

Jab tum ```helm uninstall``` karte ho, toh Helm sirf un resources ko delete karta hai jo uske Manifest mein the.
- PersistentVolumeClaims (PVCs): Inke lifecycle ko Helm control nahi karta uninstall ke waqt taaki galti se database ka data na udd jaye.
- Orphaned Resources: Disk (EBS/Azure Disk/GCP PD) cloud provider ke end par "Available" dikhayegi par koi use nahi kar raha hoga. Tumhe uske har ghante ke paise dene padenge.

To tumko ye process follow karna hai:
- Uninstall App: ```helm uninstall my-db -n prod```.
- Check Leaks: ```kubectl get pvc -n prod``` (Tumhe yahan abhi bhi purani disk dikhegi).
- Manual Purge: ```kubectl delete pvc data-my-db-postgresql-0 -n prod```.

Tabhi jaakar cloud provider wo disk release karega aur tumhara bill rukega.
