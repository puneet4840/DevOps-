# Terraform Workspace

Terraform mein workspace ek feature hota hai jo ek hi Terraform codebase ko use karke multiple environments manage karne mein help karta hai — jaise:
- dev
- staging
- prod

Matlab:
- Same Terraform code, but alag-alag state files aur alag infrastructure.

Workspace ka actual kaam hota hai Same Terraform code ko use karke alag-alag state maintain karna

Matlab:
- Code same rahega.
- Lekin infrastructure alag create hoga.
- Aur har environment ka state file alag hoga.

Jaise tum terraform script se resource banate ho aur apply karte ho to vo saara code default workspace mein hota hai. Jab tum workspace use karte ho to ek generalized terraform code likhte ho. Jisko baad mein alag-alag create karke alag-alag environments ke liye use kiya jata hai.

Workspace ko ese samjho ki same code ke liye ek alag state file maintain karna.

<br>

### Sabse Pehle Problem Samjho

Suppose tumhare paas ek simple Terraform code hai:
```
resource "azurerm_resource_group" "rg" {
  name     = "my-rg"
  location = "Central India"
}
```
Ab tumne ye apply kiya.

Terraform ek ```terraform.tfstate``` file bana dega.

Is state file mein kya hota hai?
- Kaunsa resource bana
- Resource ka ID
- Current infrastructure ka status
- Terraform mapping

<br>
<br>

### Ab Problem Aayi Multiple Environments Mein

Company bolti hai:
- Dev bhi chahiye
- QA bhi chahiye
- Prod bhi chahiye

Ab tumhare paas same code hai.

Lekin tum chahte ho:
| Environment | Resource Group |
| ----------- | -------------- |
| Dev         | dev-rg         |
| QA          | qa-rg          |
| Prod        | prod-rg        |


Agar tum same folder mein direct apply karoge:
```
terraform apply
```
To problem hogi:
- Same state file use hogi.
- Terraform confuse hoga.
- Existing infra overwrite ho sakta hai.

<br>

### Terraform Workspace Kya Karta Hai

Workspace basically:
- Alag state file maintain karta hai.
- Same code reuse karta hai.

Matlab ek workspace ek alag state file create karta hai. Jaise tum normally terraform code likhke apply karte ho to ek state file create hoti hai to vo default workspace mein create hoti hai. Agar tum ek naya workspace create karke use code ko apply karoge to us workspace ke liye ek alag state file create ho jayegi. 

To simple workspace ka kaam alag-alag environments manage karna hai same code ka use karke.

Matlab internally:
| Workspace | State File    |
| --------- | ------------- |
| default   | default state |
| dev       | dev state     |
| qa        | qa state      |
| prod      | prod state    |

Code same.

Lekin har workspace:
- apna state
- apni infra mapping
- apna environment

maintain karta hai.

<br>
<br>

### Default Workspace

Jab tum terraform init karte ho:
```
terraform init
```
Terraform automatically ek workspace banata hai:
```
default
```

Check karne ke liye:
```
terraform workspace list
```
Output:
```
* default
```

### Backend Mein Workspace Kaise Store Hote Hain

**Local Backend**:

Agar local backend hai, matlab state file local system par hi store ho rahi hai to:
```
terraform.tfstate.d/
```
Folder create hota hai. Us folder ke ander workspace folder ke ader terraform state file store hoti hai.

Example:
```
terraform.tfstate.d/
 ├── dev/
 │    └── terraform.tfstate
 ├── prod/
 │    └── terraform.tfstate
```

<br>

**Remote Backend**:

Agr remote backend hai to:

Suppose backend:
```
terraform {
  backend "azurerm" {
    resource_group_name  = "tf-backend-rg"
    storage_account_name = "tfstateprod"
    container_name       = "tfstate"
    key                  = "terraform.tfstate"
  }
}
```
```
tfstate container
│
├── terraform.tfstate                 ← default workspace
│
├── env:/dev/terraform.tfstate
│
├── env:/prod/terraform.tfstate
```

Terraform automatically:
- workspace name detect karta hai
- alag state path bana deta hai

Tum manually path create nahi karte.

<br>

**Remote Backend End-to-End Flow**:

Step 1 — Backend Configure:
```
terraform {
  backend "azurerm" {
    resource_group_name  = "tfstate-rg"
    storage_account_name = "tfstorage123"
    container_name       = "tfstate"
    key                  = "terraform.tfstate"
  }
}
```

Step 2 — Init:
```
terraform init
```
Terraform:
- backend connect karta hai
- remote state initialize karta hai

Step 3 — Create Dev Workspace:
```
terraform workspace new dev
```
Terraform internally:
```
env:/dev/terraform.tfstate
```
create/use karega.

Step 4 — Apply:
```
terraform apply
```
Now:
- dev infrastructure deploy hoga
- dev state remote blob mein save hogi

Step 5 — Switch Prod
```
terraform workspace new prod
```
Terraform internally:
```
env:/prod/terraform.tfstate
```

Final Storage Structure

Azure blob container:
```
tfstate/
│
├── terraform.tfstate
├── env:/dev/terraform.tfstate
├── env:/prod/terraform.tfstate
```
use karega.

<br>
<br>

### Ab Real Example Banate Hain

Scenario

Tumhe:
- Dev
- Prod

ke liye alag resource groups banane hain.

**Step 1 — Main.tf**:
```
provider "azurerm" {
  features {}
}

resource "azurerm_resource_group" "rg" {
  name     = "${terraform.workspace}-rg"
  location = "Central India"
}
```

**Terraform.workspace Kya Hai**:

Terraform ek built-in variable deta hai:
```
terraform.workspace
```
Ye current workspace ka naam return karta hai.

Example:
| Workspace | Value |
| --------- | ----- |
| dev       | dev   |
| prod      | prod  |

**Step 2 — Init**:
```
terraform init
```

**Step 3 — Dev Workspace Create**:
```
terraform workspace new dev
```
Apply:
```
terraform apply
```
Kya create hoga?
```
dev-rg resource group
```

**Step 4 — Prod Workspace**:
```
terraform workspace new prod
```
Apply:
```
terraform apply
```
Create hoga:
```
prod-rg resource group
```

**Yaha Actual Magic Kya Hua**:

Same code.

Lekin:
- Different workspace
- Different state
- Different infrastructure

Visualization
```
                 SAME CODE
                     │
         ┌───────────┼───────────┐
         │           │           │
       dev         qa          prod
         │           │           │
   dev.tfstate  qa.tfstate  prod.tfstate
         │           │           │
     Dev Infra    QA Infra   Prod Infra
```

**Workspace Variable Ka Advanced Use**:

Tum workspace ke basis par:
- VM size
- Instance count
- Naming
- Tags
- Scaling

sab control kar sakte ho.

Example:
```
Without Workspace

locals {
  vm_size = {
    dev  = "Standard_B1s"
    prod = "Standard_D4s_v5"
  }
}
```
Use:
```
size = local.vm_size[terraform.workspace]
```

Real Production Example
```
locals {
  environment_config = {
    dev = {
      instance_count = 1
      sku            = "B1"
    }

    prod = {
      instance_count = 5
      sku            = "P2"
    }
  }
}
```
Usage:
```
count = local.environment_config[terraform.workspace].instance_count
```

### Workspace Kab Use Karna Chaiye

Workspace tab best hota hai jab:
- Infrastructure mostly same ho.
- Small-medium setup ho.
- Team small ho.
- Temporary environments ho.
- Isolation requirements low ho.

Mainly Dev/Test environments ke liye hi Workspace ka use karna chaiye.

BEST USE CASE 1 — Dev/Test/QA Environments.

BEST USE CASE 2 — Feature Branch Testing.

BEST USE CASE 3 — Personal Sandbox.

BEST USE CASE 4 — Learning / Labs.

<br>

### Workspcae Kab Use Nahi Karna Chaiye

Workspace mainly prodcution mein avoid karna chaiye.

**BAD USE CASE 1 — Critical Production**:

Suppose:
```
prod
```
workspace mein:
- databases
- production APIs
- customer data

Agar galti se workspace switch kar diya:
```
terraform workspace select dev
```
ya:
```
terraform destroy
```

Huge disaster possible. Matlab galti se production delete ho sakte hai.

<br>

**BAD USE CASE 2 — Different Infrastructure Architecture**:

Workspace tab avoid karo jab environments structurally different hon.

Wrong Example:

Dev
```
1 VM
```

Prod
```
Kubernetes Cluster
Redis
Load Balancer
3 Databases
CDN
```
Yaha same code maintain karna messy ho jayega.

Bahut saare:
```
count
for_each
conditionals
ternary
```
aa jayenge.

Code ugly ho jayega.

<br>

Production mein workspace ki jagah tum multiple folers banakar alag-alag environments use kar sakte ho.
